# 差评监控整体流程

本文整理本项目中“差评监控”的整体业务与技术流程，便于后续维护、交接和排查问题。

---

## 1. 总体目标

差评监控的目标是：

- 定时检查当前已打开的所有浏览器标签页
- 进入每个标签页对应的点评评价后台页面
- 遍历本地配置的门店
- 筛选近 30 天的差评
- 提取差评数量和摘要信息
- 在前端界面中做红点、状态提示和弹窗提醒

它本质上是一套：**前端调度 + 主进程自动化 + iframe 内筛选抓取 + UI 通知** 的组合流程。

---

## 2. 前端调度逻辑

相关代码：`index.html:820-897`

### 2.1 开关启动

用户打开“差评监控”开关后，前端会：

1. 检查当前是否存在标签页
2. 更新状态文案为“监控中...”
3. 立即执行一次全量检查 `runBadReviewMonitorAll()`
4. 启动后续定时调度 `scheduleBadReviewMonitor()`

如果没有任何标签页，则自动关闭开关，不进入监控。

### 2.2 定时策略

相关代码：`index.html:845-864`

差评监控不是固定周期执行，而是：

- 每轮间隔在 **10~15 分钟**之间随机
- 每次执行完成后，再递归安排下一轮

这样做的目的是避免固定频率过于机械。

### 2.3 多标签页执行方式

相关代码：`index.html:866-897`

每一轮执行时，前端会：

- 收集当前所有标签页 ID
- 按顺序逐个调用 IPC：`run-bad-review-monitor`
- 每个标签页独立执行完整差评检查逻辑
- 结束后更新状态栏结果

因此，差评监控的执行粒度是：**按标签页串行执行**。

## 3. 主进程入口逻辑

相关代码：

如果未配置门店，则直接返回“请先在设置中添加门店”。

## 4. 页面导航流程

相关代码：`main.js:331-387`

主进程会在目标标签页中自动点击页面菜单，依次进入目标页面：

1. 点击“评价管理”
2. 点击“门店评价”
3. 点击“点评评价”Tab

这一步的目标是将页面切换到可进行差评筛选的业务界面。

## 5. 门店选择流程

相关代码：`main.js:388-595`

### 5.1 打开门店下拉框

系统会尝试打开门店选择面板：

- 优先在主文档中查找 `.shop-select-input`
- 如果主文档中不存在，但页面含 iframe，则继续在子 frame 中查找
- 找到后点击，展开门店列表

### 5.2 门店来源

门店来自本地设置中的门店配置，即数据库 `shops` 表中的数据。

### 5.3 遍历匹配策略

对于每个配置门店名，系统会在门店下拉列表中执行匹配：

- `name.includes(targetShop)`
- 或 `targetShop.includes(name)`

也就是说，使用的是**包含关系匹配**，不是严格全等。

### 5.4 一个配置命中多个门店项时的处理

如果同一个配置门店名在当前列表中命中多个门店项：

- 不只检查一个
- 而是对每个命中的门店项都执行一次完整流程

因此，配置层是“门店关键词/名称”，执行层是“所有命中的门店实例”。

### 5.5 门店选择结果

每次选中门店后，系统会：

1. 点击门店项对应的单选框
2. 点击底部“确定”按钮
3. 等待页面刷新完成
4. 进入评价 iframe 继续后续筛选

如果：

- 门店列表为空
- 某个配置项完全找不到可匹配门店
- 某次选择失败

则会根据场景点击“取消”关闭面板，并提前结束或跳过当前项。

## 6. iframe 内自动化逻辑

相关代码：`main.js:597-745`

评价列表和筛选区不在主文档中，而是在子 iframe 里，因此这里使用了专门的自动化辅助模块 `AutoHelper`。

相关代码：`auto-helper.js:16-214`

### 6.1 AutoHelper 的作用

`AutoHelper` 负责：

- 遍历当前页面的 frame 树
- 找到目标 iframe（这里通过 URL 包含 `shop-comment` 识别）
- 在 Electron 主进程中计算 iframe 的真实偏移坐标
- 对 iframe 内元素发送真实鼠标事件

核心能力包括：

- `findFrame(urlSubstring)`：查找目标 frame
- `getFrameOffset(frame)`：换算子 frame 相对主页面坐标
- `click(frame, target)`：发送真实点击事件
- `evaluate(frame, expression)`：在子 frame 内执行 JS
- `waitForSelector(frame, selector)`：轮询等待元素出现
- `jsClick(frame, selector)`：JS click 兜底

### 6.2 为什么不用纯 DOM click

这个页面的评价区在 iframe 里，且部分点击目标可能是跨域 iframe 中的复杂控件。

因此项目优先采用：

- 真实鼠标事件 `sendInputEvent`

而不是单纯依赖：

- `element.click()`

这样兼容性更高，更接近真实用户操作。

## 7. 差评筛选流程

相关代码：`main.js:610-685`

进入目标 iframe 后，系统会执行两步筛选。

### 7.1 Step 6：切换时间范围到“近30天”

逻辑：

- 优先查找 `.filter-button[data-id="30"]`
- 如果没有，再按按钮文案“近30天”匹配
- 优先使用 `AutoHelper.click`
- 如果失败，再回退到 `jsClick`

目标是将评价列表限制在近 30 天范围内。

### 7.2 Step 7：切换到“差评”

逻辑：

- 优先查找并点击整行 `a.grade-item`
- 通过 `data-id="3"` 或文案“差评”识别目标项
- 如果真实点击失败，再回退到 `jsClick`

这里特意优先点击整行，而不是只点某些内部小图标节点，是为了提高命中率和稳定性。

### 7.3 调试校验

相关代码：`iframe-bridge.js:174-205`

筛选后会读取调试快照信息，辅助判断：

- 差评项 DOM 是否存在
- 差评行是否像“已选中”
- `.grade-name` 是否带 `choosed` class

这部分主要用于排查页面结构变化或点击未生效的问题。

## 8. 差评列表提取逻辑

相关代码：`main.js:687-745`

完成“近30天 + 差评”筛选后，系统会在 iframe 内提取评价数据。

### 8.1 空结果判断

优先检查是否存在空态元素，例如：

- `.empty-tip`
- `.no-data`
- 或类名中包含 `empty` 的节点

如果空态可见，则认为当前门店暂无差评。

### 8.2 差评项提取

如果存在 `.comment-item`，则逐条解析：

- `reviewTime`：评价时间
- `rating`：星级
- `contentPreview`：内容预览

解析方式：

- 星级：从 `.comment-star` 的 class 中提取 `comment-rankX`
- 时间：从 `.content-date` / `.date-item` 中组合
- 内容：从 `.info-content` 或 `.comment-content` 中读取，并截断为 120 字以内预览

### 8.3 返回结果结构

每个门店会得到类似结果：

- `hasBadReviews`
- `count`
- `message`
- `reviews[]`

如果存在差评，则加入当前标签页的汇总结果中。

## 9. 标签页内的汇总逻辑

相关代码：`main.js:488-493`、`main.js:737-790`

在单个标签页的一轮检查中，系统会累计：

- `totalBadReviews`：当前标签页共发现多少条差评
- `badReviewDetails`：按门店归类的差评详情
- `totalShopRuns`：本轮实际检查了多少个门店实例

因此，汇总单位不是全局，而是：**单个标签页内的全部门店结果汇总**。

## 10. 通知与前端展示逻辑

### 10.1 主进程发通知

相关代码：`main.js:759-781`

当一个标签页检查完成后：

- 如果发现差评：
  - 发送 `bad-review-found`
  - 触发系统通知 `Notification`
- 如果没有差评：
  - 发送 `bad-review-cleared`

系统通知中会带上总条数，必要时附带一条示例信息。

### 10.2 前端收到差评通知后的处理

相关代码：`index.html:766-794`

前端收到 `bad-review-found` 后会：

1. 更新该标签页状态为存在差评
2. 记录差评数量
3. 重新渲染标签页红点
4. 更新自动化面板状态文案
5. 弹出差评提醒弹窗

### 10.3 标签页红点与弹窗

相关代码：

- `index.html:603-640`：标签页红点展示
- `index.html:707-759`：差评提醒弹窗

最终用户会看到：

- 标签页上的差评提醒红点
- 状态栏中的本轮检查结果
- 弹窗中的差评门店与条数摘要

## 11. 整体执行链路总结

可以把差评监控理解为下面这条链路：

1. 用户打开差评监控开关
2. 前端立即执行一轮全标签检查，并设置 10~15 分钟随机调度
3. 前端逐个标签调用主进程 `run-bad-review-monitor`
4. 主进程进入评价管理 → 门店评价 → 点评评价
5. 打开门店下拉框，按本地配置遍历匹配门店
6. 对每个命中的门店实例进入评价 iframe
7. 切换到“近30天”
8. 切换到“差评”
9. 提取差评列表、时间、星级、内容预览
10. 按标签页汇总所有门店结果
11. 若发现差评，则通知前端并弹窗提醒；否则清除该标签的差评标记

---

## 12. 一句话总结

差评监控的本质是：**前端定时轮询所有标签页，主进程在每个标签页里自动进入点评评价页面，遍历配置门店，在评价 iframe 中筛选近 30 天差评并提取结果，再回传前端做红点、状态和弹窗提醒。**
